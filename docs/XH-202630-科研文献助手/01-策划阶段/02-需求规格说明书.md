## 1 引言

### 1.1 编写目的

本需求规格说明书旨在明确"科研文献智能助手"系统的全部功能需求、非功能需求、数据需求及接口需求，为系统设计、开发、测试和验收提供基准依据。

本文档的预期读者包括：项目开发者、测试人员、技术报告编写者、发榜单位评审人员。

### 1.2 项目背景

科研文献分析是学术研究的基础工作，但当前面临以下核心痛点：

1. **文献量大难覆盖**：AI/Agent领域论文数量庞大，研究者难以全面阅读
2. **单篇论文阅读耗时**：每篇论文长达数十页，提取关键信息费时费力
3. **论文筛选困难**：难以快速识别必读论文和可跳过论文
4. **观点对比困难**：不同论文的观点和方法难以系统对比，观点矛盾难以发现
5. **综述撰写耗时**：文献综述通常需要数天时间，格式和引用易出错
6. **内容适配缺失**：不同知识水平的研究者获得的分析内容无差异化

本课题由上海云之脑智能科技有限公司（科大讯飞全资子公司）发榜，要求研究**领域知识个性化生成与多智能体协同决策**技术，并在至少一个垂直领域完成落地验证。

## 2 总体描述

### 2.1 系统概述

科研文献智能助手是一个基于多智能体协同的科研文献分析系统。研究者输入研究主题后，多个AI Agent自动协作完成**文献检索、内容分析、知识提取、综述生成**，并根据研究者的方向和水平输出个性化的文献分析报告。

**一句话描述**：构建一个多智能体协同的科研文献分析系统，研究者输入研究主题后，多个AI Agent自动协作完成文献检索、内容分析、知识提取、综述生成，并根据研究者的方向和水平输出个性化的文献分析报告。

### 2.2 系统用户

| 用户角色 | 描述 | 典型场景 |
|---------|------|---------|
| 本科生 | 计算机相关专业，对该领域零基础或了解较少 | 需要通俗易懂的入门指南、类比解释 |
| 硕士生 | 已有基础，正在开展研究 | 需要方法对比、实验分析、代码示例 |
| 博士生 | 深入研究某一方向 | 需要前沿分析、研究空白识别、创新建议 |
| 教师/研究者 | 备课或开展研究 | 需要知识体系梳理、教学案例、参考文献 |

### 2.3 运行环境

| 环境 | 要求 |
|------|------|
| **客户端** | 现代浏览器（Chrome 90+、Firefox 88+、Safari 14+、Edge 90+），支持移动端适配 |
| **服务端操作系统** | Linux（推荐 Ubuntu 22.04+） |
| **运行时环境** | JDK 17+、Python 3.10+、Node.js 18+ |
| **GPU要求** | 可选。软件方模型和外接API无需GPU；若使用用户本地模型（方案C）部署7B级别模型则推荐显存 ≥ 16GB，1.5B级别模型CPU即可 |
| **最低内存** | 4GB（使用软件方模型或外接API）/ 8GB（使用本地小模型）/ 32GB（使用本地7B模型，含模型加载） |
| **磁盘空间** | 10GB+（外接API模式）/ 50GB+（本地模型模式，含模型文件、论文数据、向量数据库） |
| **网络要求** | 外接API模式需稳定网络；本地模型模式无需外网 |

### 2.4 系统边界

#### 系统范围内

- 用户注册、登录与画像管理
- 基于研究主题的智能论文检索
- 单篇论文的AI深度分析
- 多篇论文的对比分析（含矛盾发现）
- 个性化文献综述生成
- Agent协同过程可视化
- 分析报告导出（PDF/Word）
- 知识溯源与引用核查

#### 系统范围外

- 论文PDF全文解析（仅支持摘要和元数据级分析）
- 实时在线论文数据库同步更新
- 用户之间的社交功能
- 论文写作辅助（仅生成综述，不辅助写作）
- 移动端原生应用（仅Web端）

| P1 | 20个 | 重要功能，显著提升用户体验 |
| P2 | 14个 | 锦上添花，时间充裕时实现 |

## 7 用户画像与个性化需求

### 7.1 画像维度定义

```
┌─────────────────────────────────────────┐
│           科研用户画像                    │
│                                         │
│  身份信息：                              │
│  ├── 学历层次：本科生/硕士生/博士生/教师  │
│  ├── 研究方向：NLP/CV/RL/多模态/...     │
│  └── 研究阶段：选题/调研/实验/写论文     │
│                                         │
│  知识水平：                              │
│  ├── 对该主题的了解程度：初级/中级/      │
│  │   高级/专家                           │
│  └── 数学/编程基础：弱/中/强             │
│                                         │
│  需求偏好：                              │
│  ├── 需求类型：入门了解/文献综述/         │
│  │   方法对比/寻找创新点                  │
│  ├── 内容深度：通俗/均衡/专业            │
│  └── 输出格式：通俗解释/学术报告/对比表格 │
└─────────────────────────────────────────┘
```

### 7.2 画像驱动的生成策略

| 用户类型 | 内容策略 | 输出特征 | 示例 |
|---------|---------|---------|------|
| 本科生·初级 | 通俗解释+类比+入门路线 | 日常用语、有比喻、推荐入门资源 | "Multi-Agent就像一个AI项目组..." |
| 硕士生·中级 | 方法对比+实验分析+代码示例 | 标准描述、有示例、结构化 | "LangGraph使用图结构定义Agent..." |
| 博士生·高级 | 前沿分析+研究空白+创新建议 | 专业术语、深入分析、引用论文 | "现有工作在实时性方面存在不足..." |
| 教师·专业 | 知识体系+教学案例+参考文献 | 系统化、有教学建议 | "教学建议：从集中式架构讲起..." |

### 7.3 个性化实现要求

1. **Prompt注入**：用户画像以结构化格式注入Agent的系统Prompt，使用醒目标注（如【重要】标签）
2. **难度适配**：根据knowledge_level字段动态调整生成内容的专业术语密度和深度
3. **风格适配**：根据preferred_style字段动态调整表达方式
4. **推荐增强**：根据research_field字段对检索结果进行二次排序
5. **对比验证**：系统必须支持同一问题、不同画像用户的输出对比（用于演示和评估）

### 10.4 实际 API 契约（与原文档的差异说明）

| 端点 | 原文档规范 | 实际实现 | 差异说明 |
|------|----------|---------|---------|
| `GET /api/users/{userId}` | 响应字段：`{userId, username, email, createdAt}` | 响应字段：`{user_id, username, email, has_profile, created_at}` | 全局 snake_case + 新增 has_profile 标识 |
| `PUT /api/users/{userId}` | 请求：`{email?, password?}` | 请求：`{username, email}` | 密码修改走独立端点（如需） |
| `GET /api/sessions` | 响应：用户历史会话 | 同 | snake_case 字段 + 状态机 active/completed/expired |
| 入参枚举 | `master` | 接受 `master` / `MASTER` | 全局 `accept-case-insensitive-enums: true` |
| `POST /api/papers/search` | `{q, yearFrom, sort}` | 6 参数 `q + yearFrom + yearTo + venue + sort + page + size` | 与原文档一致，扩展为 6 维 |
| 错误响应 | `{code, message}` | `{code, message, data, timestamp}` + HTTP 状态码对齐 | 新增 data 字段 + HTTP 状态码正确语义 |
