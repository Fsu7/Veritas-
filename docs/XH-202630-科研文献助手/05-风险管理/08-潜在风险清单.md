# XH-202630 潜在风险清单与应对策略

> **课题**：领域知识个性化生成与多智能体协同决策系统研究
> **发榜单位**：上海云之脑智能科技有限公司（科大讯飞子公司）
> **创建日期**：2026年5月23日
> **文档版本**：v1.0

---

## 风险总览

```
风险等级定义：
🔴 高风险  — 可能导致项目失败，必须提前预防
🟡 中风险  — 会影响项目质量，需要重点关注
🟢 低风险  — 影响较小，可以接受或简单处理
```

| 等级 | 数量 | 说明 |
|------|------|------|
| 🔴 高风险 | 5个 | 必须在开发前制定应对方案 |
| 🟡 中风险 | 8个 | 需要在开发过程中持续关注 |
| 🟢 低风险 | 4个 | 了解即可，必要时处理 |

## 一、🔴 高风险

### 风险1：大模型生成内容不准确（幻觉问题）

| 项目 | 内容 |
|------|------|
| **风险描述** | 大模型可能生成看似合理但实际错误的内容，尤其在专业领域知识生成中，幻觉问题会导致系统不可信 |
| **发生概率** | 高（大模型固有缺陷） |
| **影响程度** | 致命 — 评委如果发现生成内容有事实错误，会直接否定系统价值 |
| **触发场景** | 知识库中没有覆盖的问题、需要推理的复杂问题、涉及具体数据的问题 |

**应对策略：**

```
方案1：RAG强约束（必须做）
├── 生成内容必须基于检索到的知识片段
├── 在Prompt中明确要求"只使用提供的参考资料回答"
├── 如果检索不到相关知识，明确告知用户"暂无相关资料"
└── 对生成内容进行来源标注，展示引用的知识片段

方案2：审核Agent把关（必须做）
├── 审核Agent检查生成内容是否有事实错误
├── 与知识库原文比对，标记不一致之处
├── 对涉及具体数字、日期、名称的信息重点核查
└── 不确定的内容标注"请以官方资料为准"

方案3：知识库质量保障（必须做）
├── 使用权威来源的文档（教材、论文、官方文档）
├── 非开发者A负责文档审核，确保知识库内容准确
├── 建立知识更新机制，及时修正错误
└── 对知识库进行分类标注（核心知识/扩展知识）
```

---

### 风险2：多Agent协同逻辑复杂导致系统不稳定

| 项目 | 内容 |
|------|------|
| **风险描述** | 6个Agent之间的任务分配、消息传递、结果融合逻辑复杂，容易出现死锁、循环、结果丢失等问题 |
| **发生概率** | 中高 |
| **影响程度** | 严重 — 系统崩溃或输出混乱会严重影响演示效果 |
| **触发场景** | 复杂多步骤任务、Agent之间依赖关系出错、异常输入 |

**应对策略：**

```
方案1：渐进式开发（必须做）
├── 第1步：先实现单Agent（协调者 + 生成），跑通基础流程
├── 第2步：加入检索Agent，实现基础RAG
├── 第3步：加入分析Agent，增强分析能力
├── 第4步：加入审核Agent，增加质量保障
└── 每一步都充分测试后再进行下一步

方案2：设计降级机制（必须做）
├── 当多Agent协同失败时，自动降级为单Agent模式
├── 设置超时机制，单个Agent执行超过N秒则跳过
├── 设置重试机制，失败后重试1次
└── 记录失败日志，便于调试

方案3：简化协同流程（建议做）
├── 不要设计过于复杂的Agent间通信协议
├── 使用LangGraph的标准模式（Sequential / Branching）
├── Agent之间的消息格式统一为JSON
└── 避免Agent之间的循环依赖
```

### 风险3：个性化效果不明显

| 项目 | 内容 |
|------|------|
| **风险描述** | 不同用户问同一问题，系统给出的回答差异不大，评委看不出个性化的价值 |
| **发生概率** | 中 |
| **影响程度** | 严重 — 个性化是课题核心要求之一，如果效果不明显会大幅扣分 |
| **触发场景** | 用户画像信息太少、Prompt中个性化指令不够强、大模型忽略个性化指令 |

**应对策略：**

```
方案1：显式画像 + 强Prompt（必须做）
├── 用户注册时收集足够的信息（身份、专业、水平、目的、风格偏好）
├── 在系统Prompt中用醒目的方式注入用户画像
├── 使用结构化格式强调个性化要求
└── 示例：
  【重要】以下是当前用户的信息，请务必根据用户特征调整回答：
  - 用户身份：大三学生
  - 知识水平：初级
  - 回答要求：通俗易懂，多用类比，避免公式

方案2：对比演示准备（必须做）
├── 准备3-5个不同画像的测试账号
├── 针对同一问题，展示不同画像下的不同回答
├── 在答辩PPT中放对比截图
└── 这是评委最容易理解的创新点

方案3：量化评估（建议做）
├── 设计"个性化差异度"指标
├── 用不同画像的用户问同一问题，计算回答的相似度
├── 相似度越低，说明个性化效果越好
└── 在技术报告中展示数据
```

### 风险4：团队单点故障（只有1个开发者）

| 项目 | 内容 |
|------|------|
| **风险描述** | 所有代码只有1个人会，如果该成员生病、考试冲突或其他原因无法参与，项目会停滞 |
| **发生概率** | 中 |
| **影响程度** | 严重 — 可能导致项目无法按时完成 |
| **触发场景** | 开发者生病、期末考试冲突、个人事务 |

**应对策略：**

```
方案1：代码文档化（必须做）
├── 所有代码必须有清晰的注释
├── 使用AI辅助生成代码说明文档
├── 维护一个技术决策记录（为什么这样设计）
└── 确保非开发者能看懂代码的整体逻辑

方案2：AI Coding降低依赖（必须做）
├── 开发者使用AI Coding工具，代码风格标准化
├── 项目结构清晰，模块化设计
├── 使用Docker容器化部署，环境可复现
└── 即使换人接手，AI也能帮助理解代码

方案3：提前预留缓冲时间（建议做）
├── 在计划中预留2周的缓冲期
├── 核心功能在8月底前完成
├── 9月只做优化和文档
└── 即使开发者9月有冲突，系统也已完成
```

### 风险5：发榜单位需求不明确或中途变更

| 项目 | 内容 |
|------|------|
| **风险描述** | 官网公布的课题描述很简短（只有一句话），详细需求不明确；开发过程中发榜单位可能提出新的要求 |
| **发生概率** | 中高（课题描述确实很模糊） |
| **影响程度** | 严重 — 做出来的东西可能不是企业想要的 |
| **触发场景** | 初审时企业认为方向不对、答辩时评委认为偏离课题 |

**应对策略：**

```
方案1：尽早联系企业（必须做）
├── 报名后第一时间联系云之脑/科大讯飞
├── 索要详细技术需求文档
├── 询问验收标准和评审重点
├── 确认是否有指定的应用领域
└── 保留沟通记录（邮件/聊天截图）

方案2：方案确认机制（建议做）
├── 在设计阶段完成后，将技术方案发给企业确认
├── 询问"这个方向是否符合贵司预期？"
├── 根据反馈调整方案
└── 获得企业的认可相当于有了"护身符"

方案3：保持方案灵活性（必须做）
├── 系统架构设计要模块化，方便调整
├── 不要把所有时间花在一个可能被否定的方向上
├── 准备Plan B：如果企业要求变更，能快速调整
└── 关注课题描述中的关键词，不要跑题
```

## 二、🟡 中风险

### 风险6：知识库构建质量不达标

| 项目 | 内容 |
|------|------|
| **风险描述** | 知识库文档质量差、覆盖不全、格式混乱，导致RAG检索效果差 |
| **发生概率** | 中（依赖非开发者A的工作质量） |
| **影响程度** | 较大 — 直接影响系统生成质量 |

**应对策略：**
- 提供清晰的知识库构建操作指南（文档格式要求、命名规范、目录结构）
- 使用自动化工具辅助文档清洗和分块
- 建立质量检查清单（文档完整性、格式正确性、内容准确性）
- 分批构建，先构建核心知识（50篇），测试效果后再扩展

---

### 风险7：大模型API成本超预算

| 项目 | 内容 |
|------|------|
| **风险描述** | 开发和测试过程中大量调用大模型API，费用超出预算 |
| **发生概率** | 中 |
| **影响程度** | 较大 — 可能影响开发进度 |

**应对策略：**
- 优先使用软件方提供的云端模型，开箱即用，成本由软件方承担
- 软件方模型不可用时使用外接API，按量计费，费用可控
- 用户本地部署开源模型（Qwen2等）作为免费兜底方案
- API调用只在必要时使用（软件方模型不可用、对比测试、最终演示）
- 实现结果缓存，相同问题不重复调用大模型
- 预算估算：软件方模型 ¥0 + API备用 ¥200 + 用户本地模型 ¥0

---

### 风险8：系统响应速度慢

| 项目 | 内容 |
|------|------|
| **风险描述** | 多Agent协同 + RAG检索 + 大模型生成，链路较长，响应时间可能超过5秒 |
| **发生概率** | 中高 |
| **影响程度** | 较大 — 评委体验差，且是验收指标之一 |

**应对策略：**
- 使用流式输出（Streaming），让用户逐步看到生成过程
- 并行执行独立的Agent任务（如检索和分析可以并行）
- 对RAG检索结果进行缓存
- 使用本地模型减少网络延迟
- Agent工作流可视化本身就能缓解等待焦虑

---

### 风险9：评委认为"只是调API"

| 项目 | 内容 |
|------|------|
| **风险描述** | 评委认为系统只是包装了大模型API，没有技术创新 |
| **发生概率** | 中（这是所有AI应用类课题的通病） |
| **影响程度** | 较大 — 影响评分 |

**应对策略：**
- 重点展示Agent协同逻辑的设计（不是调API，是设计了协同策略）
- 展示个性化策略的实现细节
- 做充分的对比实验（vs 单Agent、vs 无个性化）
- 在技术报告中详细说明每个模块的设计思路
- 准备好回答"你的创新点是什么"这个问题

---

### 风险10：竞争激烈

| 项目 | 内容 |
|------|------|
| **风险描述** | Multi-Agent是热门方向，可能有多个团队选择类似课题 |
| **发生概率** | 中 |
| **影响程度** | 中 — 影响获奖概率 |

**应对策略：**
- 在"个性化"维度做深做透，形成差异化
- 准备高质量的演示视频和答辩PPT
- 知识库选择有特色的领域（如教育+AI，与科大讯飞契合）
- Agent协同过程可视化是独特的演示亮点

---

### 风险11：前端界面粗糙影响印象

| 项目 | 内容 |
|------|------|
| **风险描述** | 只有1个开发者，前端可能做得比较简陋，影响评委第一印象 |
| **发生概率** | 中高 |
| **影响程度** | 中 — 不影响核心评分，但影响主观印象 |

**应对策略：**
- 使用成熟的UI组件库（Element Plus / Ant Design），保证基本美观
- 用AI生成前端代码，节省时间
- 重点打磨对话界面（这是评委最关注的页面）
- Agent工作流可视化用ECharts做，技术感强
- 不需要做很多页面，做好2-3个核心页面即可

---

### 风险12：评估数据不足

| 项目 | 内容 |
|------|------|
| **风险描述** | 技术报告需要量化评估数据（准确率、对比实验等），但准备不足 |
| **发生概率** | 中 |
| **影响程度** | 中 — 影响技术报告质量 |

**应对策略：**
- 提前设计评估方案（至少50个测试问题）
- 非开发者A负责准备测试集和人工评分
- 设计自动化评估脚本
- 预留2周时间专门做评估实验

---

### 风险13：时间管理失控

| 项目 | 内容 |
|------|------|
| **风险描述** | 3.5个月时间看似充裕，但实际上要学新技术、做开发、写文档、录视频，容易失控 |
| **发生概率** | 中高（学生团队常见问题） |
| **影响程度** | 较大 — 可能导致最后匆忙提交 |

**应对策略：**
- 严格按照项目策划案的里程碑执行
- 每周开一次进度同步会（30分钟）
- 使用简单的项目管理工具（飞书/腾讯文档看板）
- 核心功能在8月底前必须完成
- 9月只做优化、文档、视频

## 三、🟢 低风险

### 风险14：技术选型不当

| 项目 | 内容 |
|------|------|
| **风险描述** | 选择了不合适的技术栈，后期难以调整 |
| **发生概率** | 低（LangChain/LangGraph生态成熟） |
| **应对** | 使用主流技术栈，遇到问题容易找到解决方案 |

### 风险15：演示环境故障

| 项目 | 内容 |
|------|------|
| **风险描述** | 答辩演示时系统崩溃或网络故障 |
| **发生概率** | 低 |
| **应对** | 准备演示视频作为备份；本地部署模型，不依赖网络 |

### 风险16：知识版权问题

| 项目 | 内容 |
|------|------|
| **风险描述** | 知识库中使用的文档可能存在版权问题 |
| **发生概率** | 低 |
| **应对** | 使用公开资料（教材、开源论文、政府文件），注明来源 |

### 风险17：团队成员积极性下降

| 项目 | 内容 |
|------|------|
| **风险描述** | 长期开发过程中团队成员积极性下降 |
| **发生概率** | 低中 |
| **应对** | 设定阶段性目标，每完成一个里程碑庆祝一下；明确获奖后的收益（奖金、实习机会） |

## 六、风险检查清单

在项目每个里程碑节点，使用以下清单检查：

```
□ 大模型生成内容是否经过审核Agent检查？
□ 多Agent协同是否有降级/超时机制？
□ 个性化效果是否可以通过对比测试验证？
□ 代码是否有完整注释和文档？
□ 发榜单位是否确认了技术方案方向？
□ 知识库文档是否经过质量审核？
□ API费用是否在预算范围内？
□ 系统响应时间是否 < 5秒？
□ 是否准备了"创新点"的答辩话术？
□ 前端核心页面是否美观可用？
□ 评估数据是否充分（50+测试用例）？
□ 项目进度是否在计划范围内？
```

---

> **文档维护**：每个里程碑节点更新风险状态（已解决/进行中/新增）
> **负责人**：非开发者B（项目管理）负责跟踪风险状态
