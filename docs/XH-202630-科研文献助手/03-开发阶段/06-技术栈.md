```sql
-- 论文表
CREATE TABLE papers (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    paper_id VARCHAR(100) UNIQUE NOT NULL COMMENT '论文唯一ID',
    title VARCHAR(500) NOT NULL COMMENT '标题',
    authors JSON COMMENT '作者列表',
    abstract TEXT COMMENT '摘要',
    year INT COMMENT '发表年份',
    venue VARCHAR(200) COMMENT '发表会议/期刊',
    keywords JSON COMMENT '关键词',
    citation_count INT DEFAULT 0 COMMENT '引用数',

    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,

    INDEX idx_year (year),
    INDEX idx_venue (venue),

) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 用户表
CREATE TABLE users (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id VARCHAR(100) UNIQUE NOT NULL,
    username VARCHAR(100) NOT NULL,
    email VARCHAR(200),
    password_hash VARCHAR(200) NOT NULL COMMENT 'BCrypt加密密码哈希',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 用户画像表
CREATE TABLE user_profiles (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    user_id VARCHAR(100) NOT NULL,
    education_level ENUM('undergraduate', 'master', 'phd', 'faculty') COMMENT '学历',
    research_field VARCHAR(200) COMMENT '研究方向',
    knowledge_level ENUM('beginner', 'intermediate', 'advanced', 'expert') COMMENT '知识水平',
    preferred_style ENUM('simple', 'balanced', 'technical') COMMENT '偏好风格',
    profile_data JSON COMMENT '完整画像数据',
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(user_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;

-- 会话表
CREATE TABLE sessions (
    id BIGINT PRIMARY KEY AUTO_INCREMENT,
    session_id VARCHAR(100) UNIQUE NOT NULL,
    user_id VARCHAR(100) NOT NULL,
    topic VARCHAR(500) COMMENT '研究主题',
    status ENUM('active', 'completed', 'expired') DEFAULT 'active',
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES users(user_id)
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4;
```

## 五、模型层技术栈（灵活配置）

> **核心设计理念**：软件方提供的模型优先级最高，确保开箱即用。系统采用**三路并行**架构，支持软件方模型、外接API、用户本地模型三种方式，按优先级自动降级切换。用户无需任何配置即可使用软件方模型，也可以选择接入自己的API或在本地部署模型。

### 5.1 大语言模型

| 方案 | 模型 | 用途 | 部署方式 | 配置要求 | 成本 |
|------|------|------|---------|---------|------|
| **方案A：软件方提供** | 软件方云端模型服务 | 主力模型（Agent推理、内容生成） | 云端API调用 | 无需配置，开箱即用 | 软件方承担 |
| **方案B：外接API** | OpenAI兼容接口 / 讯飞星火 / DeepSeek / 通义千问等 | 备用模型（用户自配） | API调用 | 需网络和API Key | 按量计费 |
| **方案C：用户本地模型** | Qwen2-7B / Qwen2-1.5B / 其他开源模型 | 兜底模型（用户本机部署） | 本地部署 | 需GPU或CPU | ¥0 |

**三路降级策略：**
```
优先级：方案A（软件方模型）> 方案B（外接API）> 方案C（用户本地模型）

启动时自动检测：
├── 默认使用方案A（软件方提供的云端模型，开箱即用）
├── 方案A不可用时，检测用户配置的API Key → 使用方案B（外接API）
├── 方案B也不可用时，检测本地模型文件 → 使用方案C（用户本地模型）
└── 运行时降级：
    ├── 方案A失败（服务异常/额度限制）→ 自动切换方案B

```

## 七、技术栈选型理由

### 为什么Java + Python混合架构？

| 考虑因素 | Java优势 | Python优势 | 结论 |
|---------|---------|-----------|------|
| **团队背景** | 开发者熟悉Java | 开发者熟悉Python | 两者都用 |
| **企业级特性** | Spring生态成熟，事务、安全、监控完善 | 相对较弱 | Java做主服务 |
| **AI/ML生态** | 较弱 | LangChain、Transformers等生态最强 | Python做AI服务 |
| **性能** | JVM性能优秀 | GIL限制，但AI推理用C++后端 | Java处理业务，Python处理AI |
| **部署** | Jar包部署简单 | 依赖管理复杂 | Docker统一解决 |

### 为什么选LangGraph而不是其他Agent框架？

| 框架 | 优势 | 劣势 | 选择理由 |
|------|------|------|---------|
| **LangGraph** | 与LangChain生态无缝集成，可视化支持好，社区活跃 | 较新，文档在完善中 | ✅ 最适合本项目 |
| **AutoGen** | 微软出品，功能强大 | 学习曲线陡峭，配置复杂 | 时间有限，不选 |
| **CrewAI** | 角色扮演概念清晰 | 功能相对简单 | 备选 |
| **MetaGPT** | 针对软件开发优化 | 不适合通用任务 | 不选 |

### 为什么软件方提供模型优先，而非强制用户本地部署？

| 方案 | 开箱即用 | 成本 | 硬件要求 | 灵活性 | 选择 |
|------|---------|------|---------|--------|------|
| 纯本地部署 | ❌ 需用户安装模型 | ¥0 | 高（需GPU） | 低（受硬件限制） | ❌ 门槛高 |
| 纯API | ✅ 配置即可 | ¥200-500 | 低（仅需网络） | 中 | ❌ 成本不可控 |
| **软件方模型优先** ✅ | ✅ **零配置即可使用** | ¥0-200 | **无最低要求** | **高（三路降级）** | ✅ 最终方案 |

**核心理念**：用户下载软件后**无需任何配置即可使用**——软件方提供的云端模型是默认最优方案。用户也可以按需接入自己的API或在本地部署模型，系统按优先级自动降级切换。

## 九、技术风险与应对
