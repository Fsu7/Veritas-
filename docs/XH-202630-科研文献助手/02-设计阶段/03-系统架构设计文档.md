## 1 架构概述

### 1.1 系统定位

科研文献智能助手是一个面向科研文献分析领域的多智能体协同系统，采用**Java后端为主 + Python AI服务**的混合架构，支持从论文检索、内容分析、对比研究到个性化综述生成的全流程AI辅助。

### 1.2 架构目标

| 目标 | 说明 | 衡量标准 |
|------|------|---------|
| **可演示性** | 系统必须能稳定演示全流程功能 | 演示脚本100%可执行 |
| **可解释性** | Agent协同过程可视化，知识可溯源 | Agent状态实时可见，引用可跳转 |
| **个性化** | 不同用户获得差异化输出 | 画像差异度 > 60% |
| **可靠性** | Agent失败不导致系统崩溃 | 降级机制成功率 > 95% |
| **可扩展性** | 新增Agent/模型不影响现有架构 | 新增Agent无需修改编排框架 |
| **开发效率** | 1名开发者+AI Coding在14周内完成 | 按里程碑准时交付 |

### 1.3 架构约束

- 团队仅1名开发者，需最大化AI Coding辅助效率
- 软件方模型优先，支持软件方模型/外接API/用户本地模型三路并行，按优先级自动降级
- 开发周期14周，架构需简洁高效
- 预算 ≤ ¥1,500，优先开源方案
- 发榜单位需求待确认，架构需保持灵活性

## 2 架构设计原则

| 原则 | 说明 | 体现 |
|------|------|------|
| **分层解耦** | 各层通过明确接口通信，互不侵入 | Java与Python通过REST API通信 |
| **渐进式开发** | 从简单到复杂，逐步叠加功能 | 先单Agent，后多Agent协同 |
| **降级优先** | 任何组件失败都有降级方案 | 本地模型→API降级，多Agent→单Agent降级 |
| **配置外置** | 环境相关配置不硬编码 | application.yml + .env |
| **AI Coding友好** | 代码结构清晰，AI辅助生成友好 | 模块化设计，统一命名规范 |
| **可视化驱动** | 核心价值通过可视化直观呈现 | Agent协同过程可视化 |

### 3.2 请求流转全景

以"生成文献综述"为例，完整请求流转如下：

```
用户点击"生成综述"
    │
    ▼
前端（Vue3）
    │ POST /api/analysis/report
    │ {topic, paperIds[], userId}
    ▼
Java后端（Spring Boot）
    │ 1. JWT鉴权
    │ 2. 查询用户画像（Redis缓存 → MySQL）
    │ 3. 构建分析请求DTO
    │ POST http://ai-service:8000/api/agent/analyze
    │ {topic, paperIds, userProfile}
    ▼
Python AI服务（FastAPI）
    │ 1. 请求参数校验
    │ 2. 构建LangGraph工作流
    │ 3. 注入个性化Prompt
    ▼
协调者Agent
    │ 1. 解析用户问题
    │ 2. 分解子任务
    ▼
检索Agent ──→ Chroma向量检索 + MySQL关键词检索
    │ 返回Top10论文
    ▼
分析Agent ──→ Qwen2-7B推理
    │ 返回结构化分析结果
    ▼
对比Agent（可选）──→ Qwen2-7B推理
    │ 返回对比表格+矛盾发现
    ▼
生成Agent ──→ Qwen2-7B推理（注入个性化Prompt）
    │ 返回个性化综述
    ▼
审核Agent ──→ 事实核查 + 引用核查
    │ 返回审核结果
    ▼
协调者Agent汇总
    │
    ▼
Python AI服务返回结果
    │ {report, agentStates, citations}
    ▼
Java后端
    │ 1. 解析响应
    │ 2. 存储分析结果（MySQL）
    │ 3. 缓存结果（Redis）
    ▼
前端
    │ 1. 展示综述内容
    │ 2. 展示Agent协同可视化
    │ 3. 展示引用溯源
```

### 4.2 关键技术决策

#### 决策1：Java + Python混合架构

| 考量 | 纯Java方案 | 纯Python方案 | Java+Python混合方案 ✅ |
|------|-----------|------------|----------------------|
| AI/ML生态 | 弱 | 强 | Python处理AI，最佳 |
| 企业级特性 | 强 | 弱 | Java处理业务，最佳 |
| 团队背景 | 熟悉 | 熟悉 | 两者都用，无学习障碍 |
| 部署复杂度 | 低 | 低 | 中等，Docker统一解决 |
| 通信开销 | 无 | 无 | HTTP REST，可接受 |

**决策理由**：Java处理业务逻辑（用户管理、论文管理、缓存）具有成熟生态；Python处理AI任务（Agent、RAG、模型推理）具有最强生态。两者通过REST API通信，各自独立部署和扩展。

#### 决策2：LangGraph作为Agent编排框架

| 框架 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| **LangGraph** ✅ | LangChain生态无缝集成，图结构编排灵活，可视化支持好 | 较新，文档在完善中 | 最适合本项目 |
| AutoGen | 微软出品，功能强大 | 学习曲线陡峭，配置复杂 | 时间有限，不选 |
| CrewAI | 角色扮演概念清晰 | 功能相对简单 | 备选方案 |
| MetaGPT | 针对软件开发优化 | 不适合文献分析场景 | 不选 |

#### 决策3：软件方模型优先的三路并行配置

| 方案 | 成本 | 配置要求 | 稳定性 | 延迟 | 结论 |
|------|------|---------|--------|------|------|
| 方案A：软件方模型 | 软件方承担 | 无需配置 | 依赖软件方服务 | 200-500ms | 最高优先级，开箱即用 |
| 方案B：外接API | ¥100-300 | 需API Key | 依赖网络 | 500ms+ | 软件方模型不可用时的备选 |
| 方案C：用户本地模型 | ¥0 | 需GPU/CPU | 完全可控 | 200ms-5秒 | 最低优先级，兜底方案 |
| **三路并行** ✅ | ¥0-200 | **可零配置** | 有降级方案 | 灵活 | **最终方案** |

**决策理由**：软件方提供的模型应作为最高优先级，确保用户下载即可使用，无需任何配置。当软件方模型不可用时，降级到用户自配的外接API；最后兜底使用用户本机部署的模型。降级链路：软件方模型 → 外接API → 用户本地模型。

#### 决策4：Chroma作为向量数据库

| 数据库 | 优势 | 劣势 | 结论 |
|--------|------|------|------|
| **Chroma** ✅ | 轻量级，开箱即用，Python原生 | 大规模性能不如Milvus | 数据量 < 10万足够 |
| Milvus | 高性能，分布式 | 部署复杂，资源占用大 | 过度设计 |
| FAISS | Meta开源，性能好 | 仅库非服务，需自建服务 | 备选 |

### 6.4 缓存策略（已落地实际配置）

**缓存分区与 Key 设计**（实际位于 `RedisConfig.java` + `RedisKeyUtil.java`）：

| 缓存区 | Key 模式 | TTL | 写入位置 | 命中场景 |
|--------|---------|-----|---------|---------|
| `userProfile` | `user:profile:{userId}` | 1 小时 | UserService.getProfile | 用户画像查询 |
| `userDetail` | `user:detail:{userId}` | 30 分钟 | UserService.getUserInfo | 用户信息查询 |
| `paperDetail` | `paper:detail:{paperId}` | 30 分钟 | PaperService.getPaperDetail | 论文详情查询 |
| `paperSearch` | `paper:search:{q_yf_yt_v_s_p_sz}` (7 维 SpEL `format('%s_%s_%s_%s_%s_%d_%d', ...)`) | 10 分钟 | PaperService.searchPapers | 论文搜索 |
| `sessionDetail` | `session:detail:{sessionId}` | 30 分钟 | SessionService | 会话详情 |
| `sessionList` | `session:list:{userId}:{status}:{page}:{size}` | 10 分钟 | SessionService | 会话列表 |

**RedisConfig 关键 Bean**（修复了 JM2 阶段 B-002 P0 缺陷）：

```java
@Bean
public GenericJackson2JsonRedisSerializer jsonRedisSerializer() {
    ObjectMapper om = new ObjectMapper();
    om.registerModule(new JavaTimeModule());                     // LocalDateTime 支持
    om.disable(SerializationFeature.WRITE_DATES_AS_TIMESTAMPS);  // ISO-8601
    om.activateDefaultTyping(                                    // 多态反序列化
        LaissezFaireSubTypeValidator.instance,
        ObjectMapper.DefaultTyping.NON_FINAL,
        JsonTypeInfo.As.PROPERTY);
    return new GenericJackson2JsonRedisSerializer(om);
}
```

**缓存注解示例**：

```java
@Cacheable(value = "userProfile", key = "#userId", unless = "#result == null")
public UserProfileResponse getProfile(String userId) { ... }

@CacheEvict(value = "userProfile", key = "#userId")
public UserProfileResponse updateProfile(String userId, ProfileUpdateRequest dto) { ... }

@Cacheable(value = "paperSearch",
    key = "T(java.lang.String).format('%s_%s_%s_%s_%s_%d_%d', " +
          "#q, #yearFrom, #yearTo, #venue, #sort, #page, #size)")
public PageResponse<PaperResponse> searchPapers(...) { ... }
```

**TTL 抖动**：RedisConfig 已配置 `±10%` 随机偏移，防雪崩。

**部署注意**：`activateDefaultTyping` 改变序列化格式，**部署到生产前必须 `FLUSHDB` 旧 Redis 缓存**，否则反序列化失败。

```
│  │  节点     │  │  节点     │  │  节点     │           │
│  └───────────┘  └───────────┘  └───────────┘           │
│                                                          │
│  边（Edge）定义：                                         │
│  START → coordinate → retrieve → analyze                │
│  analyze → (条件分支) → compare | generate               │
│      ├── 论文数 >= 2 → compare → generate                │
│      └── 论文数 < 2  → generate                          │
│  generate → review → END                                 │
│                                                          │
│  条件边（Conditional Edge）：                            │
│  - 论文数量决定是否进入compare节点                        │
│  - review结果决定是否返回generate重新生成                │
│                                                          │
│  超时控制：                                               │
│  - 单节点超时：30秒                                      │
│  - 全流程超时：120秒                                     │
│                                                          │
│  降级策略：                                               │
│  - 单节点失败：跳过该节点，继续执行                       │
│  - 多节点失败：降级为单Agent模式（仅检索+生成）          │
└──────────────────────────────────────────────────────────┘
```

### 9.4 Neo4j知识图谱Schema

```cypher
// 节点类型定义
(:Paper {
    paper_id: String,      // 关联MySQL paper_id
    title: String,
    year: Integer,
    venue: String
})

(:Method {
    name: String,          // 方法名称
    category: String       // 方法类别（架构/算法/框架等）
})

(:Concept {
    name: String,          // 概念名称
    domain: String         // 所属领域
})

(:Author {
    name: String,
    affiliation: String
})

// 关系类型定义
(:Paper)-[:USES]->(:Method)          // 论文使用了某种方法
(:Method)-[:IMPROVES]->(:Method)    // 方法间改进关系
(:Paper)-[:CITES]->(:Paper)         // 论文引用关系
(:Concept)-[:RELATED_TO]-(:Concept) // 概念关联
(:Paper)-[:BELONGS_TO]->(:Concept)  // 论文属于某概念领域
(:Author)-[:AUTHORED_BY]->(:Paper)  // 作者-论文关系
```

**图谱推理增强检索**：
- 通过USES关系扩展方法相关的论文发现
- 通过CITES关系挖掘引用链上的高影响力论文
- 通过CITES关系发现潜在的观点冲突和学术辩论
- 图谱路径作为RRF融合的第三路输入（语义+关键词+图谱）

### 9.6 Jackson 全局配置（修复 JM2 阶段 B-001/B-003 P0 缺陷）

> **触发原因**：JM2 首次审阅发现响应字段未统一 snake_case、枚举入参不区分大小写，导致 6 个端点返回 500。
> **修复策略**：在 `application.yml` 一次性全局解决，避免逐 DTO 加注解。

```yaml
spring:
  jackson:
    date-format: yyyy-MM-dd HH:mm:ss
    time-zone: Asia/Shanghai
    default-property-inclusion: non_null
    property-naming-strategy: SNAKE_CASE      # B-001 修复
    mapper:
      accept-case-insensitive-enums: true      # B-003 修复
```

**修复后效果**：
- 所有 DTO 字段自动转 snake_case（`paperId` → `paper_id`）
- 入参枚举 `master` / `Master` / `MASTER` 均接受
- HTTP 状态码与业务码对齐（`GlobalExceptionHandler` 改用 `ResponseEntity`）

### 11.3 JPA Entity 编码规范（JM2 阶段 P0 修复沉淀）

> **历史踩坑**：JM2 阶段用户管理模块首次审阅发现 2 个 P0 严重问题，全部源自 JPA Entity 错误使用 Lombok `@Data`：
> 1. 新建实体（id=null）与已持久化实体 equals 返回 false
> 2. 延迟加载字段触发额外 SQL
> 3. 循环引用导致 StackOverflow
> 4. toString 泄露 passwordHash / email 敏感字段

**强制规范（所有 Entity 必须遵守）**：

```java
@Getter
@Setter
@EqualsAndHashCode(onlyExplicitlyIncluded = true)
@ToString(exclude = {"passwordHash", "email"})  // 敏感字段不输出
@Entity
@Table(name = "users")
public class User {
    @Id
    @GeneratedValue(strategy = GenerationType.IDENTITY)
    @EqualsAndHashCode.Include           // ← 仅基于主键
    private Long id;

    @JsonProperty("user_id")
    @Column(name = "user_id", unique = true, nullable = false, length = 100)
    private String userId;
    // ... 其他字段
}
```

**DTO 继承场景**：必须用 `@SuperBuilder`（不用 `@Builder`），避免子类 builder 丢失父类字段。
**`@Convert` 注解**：Enum 字段已通过 `AbstractEnumConverter` 体系处理，Entity 上**不重复**加 `@Convert`。
