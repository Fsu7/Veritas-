## 二、算法实现的时间复杂度与空间复杂度分析

### 2.1 [CRITICAL] SSE 客户端断连取消事件循环中所有任务,造成服务雪崩

**文件**: [orchestrator.py](file:///Users/achieve/Library/Mobile%20Documents/com~apple~CloudDocs/Documents/AchiEVE_MacBook_Air/Veritas(求真)/Veritas/ai-service/app/agents/orchestrator.py) 第 496-503 行

**问题描述**: 客户端 SSE 断开触发 `asyncio.CancelledError`,代码使用 `asyncio.all_tasks()` 取消事件循环中**全部**任务,而非仅取消当前工作流内的子任务。单个客户端断开即会取消:
- 其他用户的并发 SSE 流
- `LLMService._recovery_task` 后台恢复任务
- 其他 HTTP 请求的处理器
- 任意 `asyncio.create_task` 创建的后台任务

```python
except asyncio.CancelledError:
    logger.info("Workflow cancelled by client disconnect")
    for task in asyncio.all_tasks():     # ← 当前事件循环全部任务
        if task is not asyncio.current_task():
            task.cancel()
    raise
```

**时间/空间复杂度分析**: 单次断连的取消操作复杂度 O(N),其中 N 为当前事件循环中所有任务数(可达成百上千)。被取消的任务会传播 `CancelledError`,触发级联回滚,实际放大效应为 O(N×M),M 为每个被取消任务清理所需步骤。最严重情况下,10 个并发 SSE 用户中任一断连,其余 9 个全部被强杀。

**优化建议**: 维护 `self._child_tasks: List[asyncio.Task]` 列表,仅取消该列表:
```python
for task in self._child_tasks:
    if not task.done():
        task.cancel()
self._child_tasks.clear()
```

**验证方法**: 启动两个并发 SSE 工作流,断开其中一个,断言另一个仍正常运行完成。

## 三、并发控制中锁机制的使用规范性与有效性验证

### 3.1 [CRITICAL] SessionService.createSession 幂等检查存在 check-then-act 竞态

**文件**: [SessionService.java](file:///Users/achieve/Library/Mobile%20Documents/com%7Eapple%7ECloudDocs/Documents/AchiEVE_MacBook_Air/Veritas(求真)/Veritas/backend/src/main/java/com/literatureassistant/service/SessionService.java) 第 67-83 行

**问题描述**: GET → 判断 → SET 三步非原子。两个并发请求同时读到 `existingSessionId == null`,都会调 `doCreateSession` 创建两条 Session,最后 SET 互相覆盖。

```java
String existingSessionId = redisTemplate.opsForValue().get(idempotencyKey);
if (existingSessionId != null) { ... return; }
SessionResponse response = doCreateSession(userId, request);
redisTemplate.opsForValue().set(idempotencyKey, response.getSessionId(), Duration.ofMinutes(5));
```

**锁机制验证**: `@Transactional` 保护 DB 写入但无法防止并发重复创建(两事务都看不到对方未提交的写)。Redis SET 也是非原子的"先查后写"。

**优化建议**: 使用 `setIfAbsent(idempotencyKey, placeholder, TTL)` 原子获取锁,成功者才创建,失败者按现有逻辑读已建立的 sessionId。

**验证方法**: JMeter 100 并发相同 clientToken 调 createSession,断言仅创建 1 条 Session。

### 3.2 [CRITICAL] AgentClientService 在 reactive 流中执行同步阻塞 Redis 写

**文件**: [AgentClientService.java](file:///Users/achieve/Library/Mobile%20Documents/com%7Eapple%7ECloudDocs/Documents/AchiEVE_MacBook_Air/Veritas(求真)/Veritas/backend/src/main/java/com/literatureassistant/service/AgentClientService.java) 第 99-100、240-267 行

**问题描述**: `Flux.doOnNext(event -> writeAgentStateToRedis(analysisId, event))` 在 Netty event loop 线程中调用 `redisTemplate.opsForHash().putAll()` + `expire()`,这是同步阻塞 Lettuce 操作,会拖慢整个 reactor pipeline,影响其他 SSE 连接吞吐量。

```java
Flux<AgentSseEvent> dataFlux = pythonAIClient.analyzeStream(request, lastEventId)
        .doOnNext(event -> writeAgentStateToRedis(analysisId, event));
```

**锁机制验证**: Lettuce 同步 API 在 Netty event loop 线程会阻塞 I/O 线程,导致其他连接无法及时响应。Reactor 的 backpressure 机制失效。

**优化建议**: 切换到 `Schedulers.boundedElastic()`:
```java
.doOnNext(event -> Mono.fromRunnable(() -> writeAgentStateToRedis(analysisId, event))
    .subscribeOn(Schedulers.boundedElastic()).subscribe())
```
或使用 Lettuce 异步 API (`RedisAsyncCommands`)。

**验证方法**: 单 SSE 流每秒 50 个事件时,检查其他 HTTP 请求 P99 延迟是否正常。

## 九、缓存策略有效性评估

### 9.1 [CRITICAL] SessionService.deleteSession 未失效 sessionList 用户列表缓存

**文件**: [SessionService.java](file:///Users/achieve/Library/Mobile%20Documents/com%7Eapple%7ECloudDocs/Documents/AchiEVE_MacBook_Air/Veritas(求真)/Veritas/backend/src/main/java/com/literatureassistant/service/SessionService.java) 第 214-225 行

**问题描述**: 仅 `@CacheEvict(value = "sessionState")` 失效单个会话详情缓存,未失效该用户的 `sessionList` 分页列表缓存(其他 Service 用 `CacheEvictionHelper.evictByPatternAfterCommit("sessionList::session:list:" + userId + ":*")`)。删除后用户列表中仍显示已删除会话,最长 10min。

```java
@CacheEvict(value = "sessionState", key = "#sessionId")
@Transactional
public void deleteSession(String sessionId) {
    ...
    sessionRepository.delete(session);
    log.info("Session deleted ...");
}
```

**缓存策略评估**: Cache-Aside 缺失"写后删列表缓存"步骤,导致强一致性问题。

**优化建议**: 在 `sessionRepository.delete(session)` 后追加 `cacheEvictionHelper.evictByPatternAfterCommit("sessionList::session:list:" + session.getUserId() + ":*")`。

**验证方法**: 删除 session 后立即 GET `/sessions`,断言返回列表不含已删除 sessionId。

### 9.3 [HIGH] sessionStore.analysisResults Map 无限增长,永不清理

**文件**: [sessionStore.ts](file:///Users/achieve/Library/Mobile%20Documents/com%7Eapple%7ECloudDocs/Documents/AchiEVE_MacBook_Air/Veritas(求真)/Veritas/frontend/src/stores/sessionStore.ts) 第 21、39-43、96-99 行

**问题描述**: 用户每次查看报告/回放都向 Map 写入。`cleanup()` 不清理 Map,`resetStates()` 在 agentStore。长时间使用后 Map 持有大量分析结果对象(含 citations、report 全文),内存单调增长。同时 `ref<Map>` 会让 Vue 深度代理 Map 内的每个 AnalysisResult,大对象响应式开销显著。

```typescript

async function fetchAnalysisResult(analysisId) {
  const res = await analysisApi.getResult(analysisId)
  analysisResults.value.set(analysisId, res)  // ← 只 set 不 delete
  return res
}
```

**缓存策略评估**: 前端内存缓存无 LRU 上限 + Vue deep proxy 双重放大,长时间使用导致内存泄漏。

**优化建议**: 使用 LRU 策略(保留最近 N 条);改用 `shallowRef` + `markRaw(res)` 避免 deep proxy。

**验证方法**: 连续查看 50 个报告后,Chrome DevTools Memory 抓取 heap snapshot,断言 analysisResults Map 大小 ≤ 10。

### 9.4 [HIGH] SearchView 分页快速点击导致 loading 状态错乱

### 9.5 [HIGH] ReportView 切换 analysisId 时存在请求竞态

**文件**: [ReportView.vue](file:///Users/achieve/Library/Mobile%20Documents/com%7Eapple%7ECloudDocs/Documents/AchiEVE_MacBook_Air/Veritas(求真)/Veritas/frontend/src/views/ReportView.vue) 第 79-98、174-180 行

**问题描述**: 用户从报告 A 快速跳到报告 B 时,若 A 的响应慢于 B,A 的 await 完成后会把 `result.value` 覆盖为 A 的内容。没有 AbortController 或序号校验。

```typescript
async function loadResult() {
  loading.value = true
  try {
    const res = await sessionStore.fetchAnalysisResult(analysisId.value)
    result.value = res  // ← 旧请求若晚到,会覆盖新结果
    editableContent.value = res.result?.report ?? ''
  } catch (e) { ... }
  finally { loading.value = false }
}
watch(analysisId, () => { loadResult() })  // ← 不取消上一次
```

**缓存策略评估**: 无请求竞态防护,易出现"显示错位"。

**优化建议**: 在 `loadResult` 内创建本地 AbortController,每次调用 abort 上一次;或用序号闭包对比 `if (myReqId !== latestReqId) return`。

**验证方法**: 快速从 A 跳到 B,断言最终显示 B 的内容。
