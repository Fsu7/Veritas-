## 二、算法实现的时间复杂度与空间复杂度分析

### 2.1 [CRITICAL] LLM 双重超时导致降级链路被截断

**文件**: `ai-service/app/services/llm_service.py` 第 426-465 行  
**严重级别**: CRITICAL

**问题描述**: `LLMService.generate()` 方法在首次调用超时后调用 `_fallback()`（切换 provider，可能耗时数秒），然后再次调用 `asyncio.wait_for(..., timeout=timeout)`。但此方法被 `BaseAgent.execute` 的 `asyncio.wait_for(..., timeout=30)` 包裹。总耗时可能超过 60s，远超 Agent 的 30s 超时限制。

```python
# llm_service.py 第 432-449 行
return await asyncio.wait_for(
    self.active_provider.generate(prompt, max_tokens, temperature),
    timeout=timeout,  # 30s
)
except asyncio.TimeoutError:
    await self._fallback()  # 切换 provider，耗时数秒
    return await asyncio.wait_for(
        self.active_provider.generate(prompt, max_tokens, temperature),
        timeout=timeout,  # 又 30s，总计可能 >60s
    )
```

**时间复杂度分析**: O(2T) = O(60s)，其中 T 为单次 LLM 调用超时。Agent 级别 30s 超时会在首次 30s + fallback 后触发 CancelledError，导致 Fallback 尝试被强制取消，降级失效。

**优化建议**: 为 fallback 路径使用更短的超时（如 `timeout // 2`），或在 `generate` 内部追踪总耗时并动态调整。

**验证方法**: 编写单元测试模拟首次超时场景，断言总执行时间不超过 35s。

### 2.2 [HIGH] AnalyzerAgent 并行 gather 异常处理丢失 idx 映射

**文件**: `ai-service/app/agents/analyzer.py` 第 86-93 行  
**严重级别**: HIGH

**问题描述**: `asyncio.gather(*tasks, return_exceptions=True)` 返回结果中，异常被替换为硬编码的 `(0, None, None)` 而非正确的 idx。当论文 2 分析失败时，结果字典中 idx=0 的成功结果被覆盖为 None，且论文 2 的 idx 永远不会出现在映射中。

```python
# analyzer.py 第 86-93 行
gathered_raw = await asyncio.gather(*tasks, return_exceptions=True)
gathered: List[Tuple[int, Optional[dict], Optional[str]]] = []
for item in gathered_raw:
    if isinstance(item, Exception):
        logger.warning(f"Analyze task failed in gather: {item}")
        gathered.append((0, None, None))  # BUG: 硬编码 idx=0
    else:
        gathered.append(item)
```

**复杂度分析**: 并行执行 N 个论文分析的时间复杂度从 O(N×T) 降低到 O(T)（T 为单次 LLM 调用时间），但 idx 丢失导致后续结果组装错误。空间复杂度 O(N) 无问题。

**优化建议**: 使用 `enumerate` 获取正确的 idx：
```python
for i, item in enumerate(gathered_raw):
    if isinstance(item, Exception):
        gathered.append((i, None, None))
    else:
        gathered.append(item)
```

**验证方法**: 构造 3 个论文分析任务，使第 2 个任务抛出异常，断言 `results_by_idx` 中 idx=0 的结果不为 None，idx=1 的结果为 None。

### 2.3 [MEDIUM] ECharts `deep: true` watch 在高频 SSE 推送下的性能损耗

**文件**: `frontend/src/components/agent/AgentFlowChart.vue` 第 239 行

```typescript
watch(() => JSON.stringify(props.agentStates), refreshChart)
```

**验证方法**: 在 SSE 推送频率 10 次/秒时用 Chrome DevTools Performance 面板对比帧率变化。

## 三、并发控制中锁机制的使用规范性与有效性验证

### 3.1 [CRITICAL] LLMService `active_provider` 读取未加锁 — 竞态条件

**文件**: `ai-service/app/services/llm_service.py` 第 426-465、467-517 行  
**严重级别**: CRITICAL

**问题描述**: `_fallback()` 在 `self._state_lock` 内修改 `self.active_provider`，但 `generate()` 在锁外读取 `self.active_provider`。后台恢复任务 `_recovery_loop`（每 5 分钟运行）也可能在请求执行期间修改 `active_provider`。

**竞态场景**:
1. Coroutine A 调用 `generate()`，在锁外读取 `self.active_provider = Provider1`
2. Coroutine B 的超时触发 `_fallback()`，在锁内将 `active_provider` 切换为 Provider2
3. Coroutine A 继续使用已失效的 Provider1 引用，但 Provider1 可能已处于不可用状态

```python
# 锁外读取 — 竞态风险
async def generate(self, prompt, max_tokens, temperature):
    ...
    return await asyncio.wait_for(
        self.active_provider.generate(prompt, max_tokens, temperature),  # 无锁读取
        timeout=timeout,
    )
```

**优化建议**: 在 `generate` 中先在锁内读取并保存 provider 引用，然后使用本地引用：
```python
async with self._state_lock:
    provider = self.active_provider
result = await asyncio.wait_for(
    provider.generate(prompt, max_tokens, temperature),
    timeout=timeout,
)
```

**验证方法**: 编写并发测试，同时触发 fallback 和 generate，使用 mock provider 验证 generate 使用的是 fallback 前的 provider 引用。

### 3.2 [HIGH] BaseAgent.execute 不清除上次执行的 state.error

**文件**: `ai-service/app/agents/base.py` 第 71-106 行  
**严重级别**: HIGH

**问题描述**: 在 Orchestrator 的重试循环中，同一 Agent 实例可能被多次执行（如 Generator 重试）。`execute()` 设置 `status=RUNNING` 和 `started_at`，但不清除 `self.state.error`。如果第一次执行失败（设置 error），第二次执行成功，`state.error` 仍保留第一次的错误信息。

```python
async def execute(self, input_data: dict, context: dict) -> dict:
    self.state.status = AgentStatus.RUNNING
    self.state.started_at = datetime.now()
    # 缺少: self.state.error = None
```

**优化建议**: 在 execute 开头清除上一次状态：
```python
self.state.status = AgentStatus.RUNNING
self.state.started_at = datetime.now()
self.state.error = None
self.state.intermediate_result = None
```

### 3.3 [HIGH] 401 拦截器并发请求重复处理

**文件**: `frontend/src/api/index.ts` 第 69-103 行  
**严重级别**: HIGH

**问题描述**: Token 过期时，多个并发请求同时收到 401，每个请求都会设置 `isManualLogout = true` 并调用 `logout()`，导致多次 `router.push('/login')` 和多次 `ElMessage.error`。

**优化建议**: 添加防重入标志：
```typescript
let isRefreshing = false
if (error.response?.status === 401 && !isAuthRequest(error.config?.url)) {
  if (!isRefreshing) {
    isRefreshing = true
    await userStore.logout()
    router.push('/login')
    ElMessage.error('登录已过期，请重新登录')
    isRefreshing = false
  }
  return Promise.reject(error)
}
```

### 3.4 [MEDIUM] TTLCache 非线程安全 — 缓存击穿风险

**文件**: `ai-service/app/core/cache.py` 第 11-14 行  
**严重级别**: MEDIUM

**问题描述**: `cachetools.TTLCache` 不保证线程安全。缓存操作不是原子的——在 `search` 方法中，先检查缓存未命中，然后执行搜索，最后写入缓存。如果两个相同查询的请求同时到达，会执行两次搜索（缓存击穿）。

**优化建议**: 使用 `asyncio.Lock` 保护缓存写入，或使用 single-flight 模式。

### 3.5 [MEDIUM] 搜索请求竞态条件 — 无请求取消机制

**文件**: `frontend/src/stores/paperStore.ts` 第 55-78 行  
**严重级别**: MEDIUM

**问题描述**: `searchPapers` 没有取消前一个请求的机制。快速连续搜索时，旧请求可能在新请求之后返回，旧结果覆盖新结果。

**优化建议**: 使用 `AbortController` 取消前一个请求：
```typescript
let searchAbortController: AbortController | null = null
async function searchPapers(query: string, page: number = 1) {
  searchAbortController?.abort()
  searchAbortController = new AbortController()
  const res = await paperApi.search({...}, { signal: searchAbortController.signal })
}
```

## 七、数据库查询优化检查（N+1 问题）

### 7.1 [HIGH] AnalysisService.comparePapers — for 循环逐个查询论文 (N+1)

**文件**: `backend/src/main/java/com/literatureassistant/service/AnalysisService.java` 第 139-142 行  
**严重级别**: HIGH

**问题描述**: `comparePapers` 方法在 for 循环中逐个查询论文，产生 N+1 查询问题。

```java
// 第 139-142 行
for (Long paperId : request.getPaperIds()) {
    Paper paper = paperRepository.findById(paperId)
        .orElseThrow(() -> new ResourceNotFoundException("论文不存在: " + paperId));
    papers.add(paper);
}
```

**复杂度分析**: N 篇论文 → N 次 SQL 查询。时间复杂度 O(N)，但每次查询涉及数据库往返延迟，实际总耗时 = N × (网络往返 + 查询执行)。

**优化建议**: 使用批量查询：
```java
List<Paper> papers = paperRepository.findAllById(request.getPaperIds());
if (papers.size() != request.getPaperIds().size()) {
    throw new ResourceNotFoundException("部分论文不存在");
}
```

**验证方法**: 开启 Hibernate SQL 日志 (`spring.jpa.show-sql=true`)，调用 `comparePapers` 传入 5 个 paperId，确认只产生 1 条 SQL 而非 5 条。

### 7.2 [HIGH] AgentClientService — SSE 流无限心跳合并导致连接永不终止

**文件**: `backend/src/main/java/com/literatureassistant/service/AgentClientService.java` 第 122 行  
**严重级别**: HIGH

**问题描述**: `generateReportStreamWithHeartbeat` 使用 `Flux.merge` 合并了无限的心跳流（`Flux.interval(Duration.ofSeconds(10))`），导致 SSE 流永不终止，连接池泄漏。

## 八、接口幂等性验证

### 8.1 [HIGH] AnalysisController.analyzePaper — 分析失败时幂等锁不释放

**文件**: `backend/src/main/java/com/literatureassistant/controller/AnalysisController.java` 第 75-93 行  
**严重级别**: HIGH

**问题描述**: 使用 `IdempotencyUtil.tryAcquireLock` 获取幂等锁（TTL 5 分钟），但分析失败时锁不会提前释放，用户 5 分钟内无法重试。

```java
// 第 75-93 行
if (!idempotencyUtil.tryAcquireLock(idempotencyKey, 300)) {
    throw new BusinessException(ErrorCode.DUPLICATE_REQUEST, "请勿重复提交，5分钟后再试");
}
try {
    // ... 分析逻辑
} catch (Exception e) {
    // 幂等锁未释放，用户需等待 5 分钟
    throw e;
}
```

**优化建议**: 在 catch 块中释放锁：
```java
catch (Exception e) {
    idempotencyUtil.releaseLock(idempotencyKey);
    throw e;
}
```

**验证方法**: 发起分析请求使其失败（如传入无效 paperId），立即再次发起请求，确认第二次请求不被幂等锁拦截。

### 8.2 [HIGH] 无 API 速率限制 — LLM 配额耗尽风险

**文件**: `ai-service/app/api/endpoints/agent.py` 第 99-200 行  
**严重级别**: HIGH

**问题描述**: `/api/agent/analyze` 和 `/api/agent/analyze/stream` 端点无速率限制。每次请求触发 6 个 Agent 的 LLM 调用，可能包含重试（最多 2×Generator + 2×Reviewer）。恶意用户可通过并发请求耗尽 LLM API 配额。`RateLimitException` 已定义但从未使用。

**优化建议**: 添加基于 Redis 或内存的速率限制中间件，限制每用户/每 IP 的并发分析请求数。

### 8.3 [MEDIUM] comparePapers / generateReport 接口幂等性依赖前端去重

**文件**: `backend/src/main/java/com/literatureassistant/controller/AnalysisController.java`

### 8.4 [MEDIUM] SSE Token 通过 URL 查询参数传递 — 安全风险

## 九、缓存策略有效性评估

### 9.1 [HIGH] UserService.updateProfile — 更新后未失效 Spring Cache

**文件**: `backend/src/main/java/com/literatureassistant/service/UserService.java` 第 222-242 行  
**严重级别**: HIGH

**问题描述**: `updateProfile` 方法更新用户画像后，没有失效 `@Cacheable` 缓存的画像数据，导致最长 1 小时返回陈旧数据。违反了 AGENTS.md 规定的"Cache-Aside — 写后删 Redis"原则。

```java
@Cacheable(value = "userProfile", key = "#userId")
public UserProfile getProfile(Long userId) { ... }

@Transactional
public void updateProfile(Long userId, ProfileUpdateRequest request) {
    // ... 更新逻辑
    // 缺少: @CacheEvict(value = "userProfile", key = "#userId")
}
```

**优化建议**: 在 `updateProfile` 方法上添加 `@CacheEvict`：
```java
@CacheEvict(value = "userProfile", key = "#userId")
@Transactional
public void updateProfile(Long userId, ProfileUpdateRequest request) { ... }
```

**验证方法**: 调用 `getProfile` 获取画像，调用 `updateProfile` 更新画像，立即调用 `getProfile`，确认返回的是更新后的数据。

### 9.2 [HIGH] SearchService 缓存未区分 user_profile — 个性化结果串扰

**文件**: `ai-service/app/services/search_service.py` 第 124-154 行

### 10.3 架构亮点

本次检查也发现了项目中值得肯定的设计：

1. **三级降级策略完善**：每个 Agent 都有 LLM → 规则的降级路径，单个 Agent 失败不阻塞工作流
2. **并行化优化**：AnalyzerAgent 使用 Semaphore + gather 并行化 LLM 调用
3. **Token 爆炸防护**：ComparerAgent 和 GeneratorAgent 都有 `_truncate_analysis_for_prompt` 截断
4. **多 Provider Embedding 架构**：支持 DashScope / Jina / OpenAI 三路降级
5. **SSE 流式推送**：支持 10 种事件类型、keep-alive ping、Last-Event-ID 断线重连
6. **缓存 TTL 随机抖动**：防止缓存雪崩，`sync=true` 防止缓存击穿
7. **数据隔离校验上移**：在 Controller 层统一校验用户数据访问权限
8. **事务边界拆分**：AI 调用在事务外执行，避免长事务
9. **JWT 安全实现**：Token 黑名单机制、密钥长度校验、Token 掩码日志
10. **Markdown 安全配置**：`html: false` 禁用原始 HTML，tooltip 使用 escapeHtml
